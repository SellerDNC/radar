Radar — Inteligência de Mercado com IA
Em 30 segundos, o seller sabe se vale entrar em um nicho, por que, e o que fazer primeiro. O Radar combina dados reais da API do Mercado Livre com inteligência artificial para entregar o que hoje leva 3-4 horas de trabalho manual.
O problema que resolve
Sellers do ML passam horas pesquisando nichos manualmente — contando vendedores, anotando preços, tentando estimar volume. Nenhuma ferramenta existente interpreta os dados: todas mostram números, mas ninguém entrega um veredicto claro com justificativa.
Proposta de valor única: A IA não só mostra dados — ela interpreta e entrega: "Vale entrar neste nicho porque... mas cuidado com..." — com score, faixa de preço vencedora e plano de ação.

Módulos do Radar
6 módulos integrados. O Radar de Mercado é o core — os outros são diferenciais complementares que retêm o seller na plataforma.
1. Radar de Mercado (core)
Seller digita uma keyword. Sistema busca dados reais (API pública) + dados da conta (OAuth). IA processa e entrega em menos de 15 segundos:
1
Volume do mercado
Total de anúncios ativos, sellers únicos, itens vendidos estimados
2
Faixa de preço vencedora
Onde está concentrado o volume de vendas. Min/Max/Média/Moda.
3
Nível de concorrência
Baseado nos parâmetros configuráveis de cada seller (ver aba Parâmetros)
4
Seu posicionamento
Se você já vende no nicho: sua posição vs top 10, seu preço vs mercado
5
Veredicto com justificativa
Vale entrar / Oportunidade com ressalvas / Evite — com texto explicando o porquê
2. Diagnóstico de Anúncio
Score 0-100 com plano de ação prioritizado. Analisa título, fotos, ficha técnica, preço e envio vs concorrentes reais.
3. Calculadora de Precificação
Clássico/Premium, DAS por faixa RBT12, ADS%, frete rateado, ICMS-ST, PDV sugerido e simulador de cenários.
4. Financeiro PJ vs PF
Separação real entre empresa e pessoal. Nenhum concorrente faz isso. Dashboard dual com pro-labore e fluxo de caixa.
5. Calendário do Seller
Obrigações MEI automáticas + campanhas ML sazonais pré-carregadas. Vencimentos com cores por área.
6. Gerador de Títulos ML
Keywords reais da API pública. 8 variações otimizadas com score de relevância estimado.
Worker ML Multi-Tenant
O Worker é o coração da integração ML. Precisa ser reescrito do zero para suportar qualquer seller — não apenas a conta do André. Roda no Cloudflare Workers Paid ($5/mês), eliminando o timeout de 30s.
⚠️ Primeiro passo crítico: Sem o Worker multi-tenant funcionando, nenhum outro módulo que dependa de dados reais do ML funciona.
Estrutura do Worker
// Endpoints necessários GET /auth → inicia fluxo OAuth (redirect para ML) GET /callback → troca code por tokens e salva no Supabase POST /refresh → renova access_token usando refresh_token GET /orders → pedidos do seller (chunked, 30 dias por vez) GET /listings → anúncios do seller (lista com paginação) GET /market → busca API pública ML (sem auth necessária) GET /me → dados do perfil do seller GET /item/:id → dados completos de um anúncio específico
Passo a passo — criar o Worker
1
Criar Worker no Cloudflare
dash.cloudflare.com → Workers & Pages → Create → nome: radar-ml-worker
Upgrade para Paid ($5/mês) antes de publicar
2
Variáveis de ambiente (secrets)
Settings → Variables → adicionar:
ML_CLIENT_ID = seu client_id do DevCenter
ML_CLIENT_SECRET = seu client_secret
SUPABASE_URL = https://zyidoasvvutddutyezwj.supabase.co
SUPABASE_SERVICE_KEY = (service role key, NÃO a anon key)
3
Adicionar redirect URI no ML DevCenter
developers.mercadolivre.com.br → seu app → Editar → URIs de redirect → adicionar:
https://radar-ml-worker.seu-user.workers.dev/callback
4
Lógica de refresh automático
Antes de cada chamada ML, verificar se expires_at < agora + 30min. Se sim, chamar /oauth/token com grant_type=refresh_token antes de prosseguir.
5
Chunked sync de pedidos
Para buscar 90 dias: dividir em chunks de 30 dias. Cada chunk = 1 chamada ao Worker. Salvar no Supabase incrementalmente. Worker Paid = sem timeout.

API Pública Mercado Livre
Dados do mercado disponíveis SEM autenticação. Qualquer busca, qualquer nicho, sem OAuth. É o que alimenta o Radar de Mercado para todos os sellers — conectados ou não.
✅ Zero atrito: Esta API não requer token, OAuth nem cadastro. Qualquer seller consegue ver dados de mercado antes mesmo de criar uma conta no Radar.

OAuth ML — 1 Clique para Qualquer Seller
O mesmo app registrado no ML DevCenter serve para qualquer seller autorizar. 1 clique → ML pede autorização → seller aceita → Radar tem acesso aos dados privados.
Fluxo completo passo a passo
1
Seller clica "Conectar Mercado Livre"
Frontend redireciona para o Worker /auth?seller_id={uuid_do_seller}
2
Worker redireciona para página oficial do ML
auth.mercadolivre.com.br — página que o seller já conhece e confia
3
Seller faz login e clica "Permitir"
ML redireciona de volta para o Worker /callback com ?code=XYZ&state=seller_id
4
Worker troca o code por tokens
POST para api.mercadolibre.com/oauth/token com client_id + secret + code
5
Tokens salvos no Supabase
{ seller_id, ml_user_id, access_token, refresh_token, expires_at }
6
Seller volta para o Radar com dados carregados
Worker redireciona para radar.com.br?connected=true — badge "ML Conectado" aparece
7
Refresh automático a cada 6 horas
Antes de qualquer chamada ML, Worker verifica se expires_at está próximo. Se sim, renova automaticamente. Seller nunca precisa reconectar.

Supabase Setup Completo
Banco de dados multi-tenant. Cada seller tem seus próprios dados isolados por seller_id. RLS garante que um seller nunca acessa dados de outro.
SQL — criar todas as tabelas
-- Sellers (auth gerenciado pelo Supabase Auth) CREATE TABLE radar_sellers ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), email TEXT UNIQUE NOT NULL, plan TEXT DEFAULT 'free', -- free | starter | pro plan_expires_at TIMESTAMPTZ, created_at TIMESTAMPTZ DEFAULT NOW() ); -- Tokens ML por seller CREATE TABLE radar_tokens ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), seller_id UUID REFERENCES radar_sellers(id), ml_user_id BIGINT, access_token TEXT NOT NULL, refresh_token TEXT NOT NULL, expires_at TIMESTAMPTZ NOT NULL, updated_at TIMESTAMPTZ DEFAULT NOW() ); -- Pesquisas do Radar de Mercado CREATE TABLE radar_pesquisas ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), seller_id UUID REFERENCES radar_sellers(id), keyword TEXT NOT NULL, resultado JSONB, veredicto TEXT, -- vale | ressalvas | evite score INTEGER, created_at TIMESTAMPTZ DEFAULT NOW() ); -- Financeiro (PJ e PF separados) CREATE TABLE radar_contas ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), seller_id UUID REFERENCES radar_sellers(id), area TEXT NOT NULL, -- pj | pf tipo TEXT NOT NULL, -- pagar | receber desc TEXT, valor NUMERIC, data DATE, status TEXT DEFAULT 'pendente', recur TEXT, -- mensal | semanal | anual created_at TIMESTAMPTZ DEFAULT NOW() ); -- Parâmetros do veredicto por seller CREATE TABLE radar_params ( seller_id UUID PRIMARY KEY REFERENCES radar_sellers(id), params JSONB NOT NULL DEFAULT '{}', updated_at TIMESTAMPTZ DEFAULT NOW() ); -- RLS: cada seller vê só seus dados ALTER TABLE radar_tokens ENABLE ROW LEVEL SECURITY; ALTER TABLE radar_pesquisas ENABLE ROW LEVEL SECURITY; ALTER TABLE radar_contas ENABLE ROW LEVEL SECURITY; ALTER TABLE radar_params ENABLE ROW LEVEL SECURITY; CREATE POLICY seller_own ON radar_tokens FOR ALL USING (seller_id = auth.uid()); -- Repetir para as outras tabelas
Parâmetros do Veredicto
Cada seller tem premissas diferentes. Um seller que trabalha com produtos de R$200+ tem critérios totalmente diferentes de um que vende por R$30. Configure os parâmetros abaixo e o Radar vai calibrar o veredicto para o seu perfil.
Como funciona: O Radar calcula um score de 0-100 baseado em critérios ponderados. Cada critério tem um peso e um threshold. Se o mercado passa do threshold, o critério contribui positivamente para o score. O veredicto final é gerado pela IA com base no score total.
📊 Critérios de Volume e Demanda
Define o que é um mercado com demanda real suficiente para justificar entrar
anúncios ativos na categoria (abaixo = mercado pequeno demais)
unidades/mês. Valida que há demanda real no nicho
% do score total (todos os pesos devem somar 100)
% de crescimento nos últimos 30 dias (0 = não considera)
⚔️ Critérios de Concorrência
Define o nível de concorrência que você está disposto a enfrentar
acima disso = mercado saturado no veredicto
% acima disso = difícil competir sem Full
de 5.0 — sellers com rep alta são mais difíceis de superar
% do score total
💰 Critérios Financeiros
Define a viabilidade econômica do nicho para o seu negócio
abaixo disso = difícil ter margem após taxas ML
margem líquida mínima para o nicho ser viável
seu faturamento anual para cálculo correto do Simples
% do score total
🏆 Thresholds do Veredicto
Define a pontuação mínima para cada classificação do veredicto final
pontos de 100 — recomendado: 60-70
pontos de 100 — entre este e o acima = amarelo
abaixo do score médio = vermelho
% do score total (fotos, fichas, títulos dos top sellers)
Preview do veredicto
Vale entrar≥ 65 pontos
Oportunidade com ressalvas40-64 pontos
Evite< 40 pontos
